# S4 1080p60 稳定版本

## 版本定位

本分支整理自 2026-07-15 实际编译、部署和测试的独立源码副本。S4 的目标是在 1080p60 视频持续编码推流的同时，把 YOLO11n 推理与视频线程解耦，避免 NPU 推理阻塞视频输出。

- Git 分支：`s4-stable-1080p60`
- 固定标签：`s4-stable-2026-07-16`
- 正式测试二进制：`imx415_yolo_rtsp_1080p60_scripted_test`
- 二进制 MD5：`fbc5c50fc6fc1b76482353a0b430d9cc`
- 二进制 SHA-256：`b76579d2324ef8bd8678ef0276775bc66d6c7181c0fff1793c93be745431605e`
- YOLO11n RKNN 模型 MD5：`91faf3f5526db7ecfed3a61a99a3ef75`

## 固定链路配置

- IMX415/V4L2：1920×1080、NV12、60 fps、4 个 mmap/DMA-BUF。
- YOLO 输入：RGA 转为 640×640 RGB888，INT8 RKNN 零拷贝输出。
- 推理策略：每 5 帧触发一次；仅当 NPU 空闲时提交最新帧；视频帧复用最近完成的检测结果。
- MPP H.264：1080p60、CBR 4 Mbps、2–6 Mbps、GOP 60、Baseline Level 4.1、CAVLC。
- RTSP：`rtsp://192.168.1.20:8554/live`；正式测试使用 RTP/RTSP interleaved over TCP。
- 频率：CPU performance governor；NPU userspace governor，600 MHz。

## 异步实现

采集线程持续获取 V4L2 帧并把最新帧交给视频线程。视频线程每 5 帧形成一次推理触发：若 NPU 空闲，复制当前 NV12 到有界推理缓冲并立即继续叠加、编码和推流；若 NPU 忙，则不排队旧任务。独立 AI 线程完成 RGA 预处理、`rknn_run` 和后处理，并在互斥锁保护下原子更新最新检测结果。

因此，检测结果约 11.75 次/秒更新，但视频输出不需要等待对应帧的推理完成，仍可保持约 58.75 fps。

## 编译

在配置好 Rockchip sysroot、RKNN Model Zoo、RGA 和 MPP 的 Ubuntu 18.04 交叉编译环境中执行：

```bash
cd yolo_integration
make clean
make TARGET=imx415_yolo_rtsp_1080p60_scripted_test
```

`Makefile` 保留了原开发机的 SDK 绝对路径，迁移环境时应先调整对应变量。重新编译后必须重新记录二进制 MD5；如果要运行自动化测试，还要同步更新 `tests/performance/scenarios.json` 中的 `expected_binary_md5`。

## S4 三轮实测

固定画面下完成 3 轮、每轮 300 秒，三轮均为 `PASS`。下列数字是三轮均值：

| 指标 | 结果 |
|---|---:|
| V4L2 采集 | 60.000 fps |
| 板端编码输出 | 58.746 fps |
| RTSP 客户端窗口 | 58.624 fps |
| 检测结果更新 | 11.748 fps |
| 视频丢帧率 | 2.091% |
| 最大推理队列 | 1 |
| 框年龄 P95 | 150.301 ms |
| 推理排队 P95 | 2.175 ms |
| `rknn_run` P95 | 58.403 ms |
| 采集到编码输出 P95 | 14.902 ms |
| 进程 CPU | 32.801% |
| NPU 平均负载 | 49.033% |
| RSS 峰值均值 | 72.70 MiB |
| 三轮最高温度中的最大值 | 62.222 °C |

`采集到编码输出` 是板内同一个 `CLOCK_MONOTONIC` 下的内部延迟，不是客户端屏幕端到端延迟。

## RTSP 修复

本版本包含两处与可播放性直接相关的修复：

1. 非阻塞 TCP socket 暂时写不完时保留当前 RTP 队列游标，避免同一帧的后续分片被下一帧覆盖。
2. RTP 时间戳按 90 kHz 与 60 fps 的有理数步进，并每秒强制周期 IDR，保证晚加入客户端能够取得解码参考帧。

## 自动化测试

脚本位于 `tests/performance/`。在能够 SSH 到板端且安装 FFmpeg 的 Ubuntu 控制机上运行：

```bash
python3 tests/performance/run_benchmark.py \
  --scenario S4 --duration 300 --repeat 3 \
  --output tests/performance/results/
```

只运行离线单元测试：

```bash
python3 -m unittest discover -s tests/performance/tests -v
```

## 已知限制

- S5 只完成一轮 1800 秒正式测试，没有完成三轮长稳重复性验收。
- S5 中吞吐、队列、CPU、NPU 和温度稳定，但 RSS 呈约 1.185 MiB/min 的持续增长；原因尚未定位，不能声称长期内存稳定。
- 正式客户端使用 RTSP-over-TCP；UDP 诊断未通过，不能声称 UDP 长时稳定。
- 没有跨设备同步时钟或可见时间码，因此客户端屏幕端到端延迟为 `Unavailable`。
- 当前 4 Mbps Baseline/CAVLC 偏向低复杂度和低带宽，1080p60 复杂画面的清晰度可能不足；提升码率需要修改 `rtsp_streaming/mpp_encoder.c` 并重新编译、复测。
