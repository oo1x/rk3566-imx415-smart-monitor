# 性能测试设计理解

## 测试真正要分开的三件事

60 fps 视频、每秒完成多少次新检测、检测框有多旧，是三件不同的事。异步设计可以让视频保持接近 60 fps，但如果 NPU 每秒只能完成十几次推理，新检测率仍只有十几 fps；在输出帧上重复画旧框不会把检测率变成 60 fps。因此报告分别使用 `processed`、`inference` 和 `detection_source_frame_id` 计算。

## S0：视频链路基线

S0 关闭 AI，用来回答相机实际 DQBUF 是否达到 60 fps、RGA Overlay/MPP/RTSP 本身能输出多少、CPU/RSS/温度和视频帧替换是多少。后续场景与 S0 对比，才能判断代价来自 AI 还是视频链路。已经完成的 S0 表明采集稳定为 60.000 fps，板端输出均值 59.011 fps；这不是“配置 60”推导出来的，而是 900 秒共 54,000 个采集事件和 53,110 个处理事件实测。

## S1：推理吞吐上限与过载证据

S1 每个视频帧都产生推理尝试。其目的不是获得可用产品策略，而是给 NPU 最大压力，观察完成率、单次各阶段耗时以及请求生成速度明显大于完成速度时的队列增长。达到安全上限就停止是正确结论，不应该为了跑满 60 秒而让内存继续增长。

## S2：固定每三帧提交是否仍会积压

60 fps 下每三帧约每秒 20 次触发。如果单次推理约 75 ms，理论服务能力约 13.3 次/秒，那么 20 次/秒的到达率仍可能积压；但这只是待验证假设，正式结论必须看真实队列和完成时间。S2 保留真实排队任务，专门显示过期任务和框年龄风险。

## S3：每三帧检查、只在空闲时取最新帧

S3 的到达检查频率仍高，但忙时不排队。NPU 空闲的那个时刻只接收最新视频帧，因此队列有界、不会处理很旧的积压帧。代价是大量推理尝试会记为 `npu_busy`，但这属于主动跳过，不是视频丢帧。最近一次完成的框继续叠加，视频输出与检测更新解耦。

## S4：降低检查频率

S4 每五帧检查一次，比较 S3 是否做了过多无效忙碌检查，以及更低触发频率能否在检测更新率、CPU、框年龄之间取得更好平衡。若 NPU 本来经常忙，S3 与 S4 的实际完成率可能接近；若 S4 让 NPU 出现较多空闲，框会更旧。必须用三轮均值和范围选择，不能只看最好一轮。

## S5：最终策略长稳

S5 只在 S3/S4 选型后执行，至少 30 分钟，重点从瞬时性能转向趋势：温度是否继续上升、RSS/PSS 是否增长、队列是否保持上界、输出和检测更新是否出现阶段性下降、网络或视频丢帧是否累积。当前配置中的 S3 参数只是候选占位，正式 S5 前应记录选型依据。

## 延迟边界

板内 `capture_ts → overlay_end` 和 `capture_ts → encode_output` 共用 `CLOCK_MONOTONIC`，无需跨机同步。框年龄也在板内计算。FFmpeg 客户端帧数能证明拉流吞吐和解码稳定，但不能直接给出“镜头实景到屏幕显示”的绝对延迟；后者需要跨端时钟同步或可见时间码/高速摄影，未实测时必须标为未完成。

## A/B 为什么独立短测

详细埋点开关、NAL 检索和 memcpy/DMA-BUF 会改变 CPU 或 I/O 行为，只能在输入、频率、编码参数和时长相同的独立短场景比较。`perf` 的函数采样比例也不能与正式低干扰 FPS 同时采集，更不能把“某函数热点占比下降”误写成“进程 CPU 同比例下降”。
