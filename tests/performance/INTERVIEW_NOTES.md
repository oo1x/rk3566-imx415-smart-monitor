# 面试复述与追问笔记

## 30 秒版本

我把 RK3566 的 IMX415→V4L2→RGA→YOLO11n/RKNN→Overlay→MPP H.264→RTSP 链路拆成视频吞吐、检测吞吐和结果新鲜度三条指标。逐帧使用同一单调时钟埋点，系统侧每秒采进程/线程 CPU、RSS/PSS、NPU、温度和队列。一条命令完成预检、预热、拉流、停止、清理、统计和三轮汇总。S1/S2 用真实有界队列证明积压，S3/S4 用仅空闲提交最新帧避免过期任务；视频丢帧和推理主动跳过始终分开。

## 常见追问

### 为什么不能说输出 60 fps 就等于检测 60 fps？

异步链路会在多个视频帧上复用同一检测结果。输出 FPS 由编码/RTSP 帧计数得到，新检测率只数完成的 inference 事件，二者来源不同。

### P95 怎么算？

所有有效原始耗时排序，使用最近秩 `ceil(0.95×N)-1`。报告同时保留样本数、平均、P50/P95/P99/最大，不能只保留成功或较快样本。

### CPU 100% 是整机满载吗？

不是。本报告进程 CPU 以一个逻辑核满载为 100%，公式是进程 user+system tick 增量除以 `CLK_TCK` 和时间。多线程进程可能超过 100%。函数 `perf` 采样占比又是另一种分母，不能混写。

### NPU load 节点读不到怎么办？

先探测 RKNN debugfs；不可读就写 `Unavailable`，频率仍单独记录。不能把固定显示 `100@600000000Hz` 的 devfreq 文件当作 100% 利用率，也不能按推理时间估算一个假负载。

### S1/S2 为什么会提前结束？

它们就是过载实验。每个 1080p NV12 帧约 3.11 MiB，无界队列既不安全也没有产品意义。脚本允许真实排队到明确上限，记录每次深度和 `queue_limit_stop` 后退出；只有捕获到这个证据才标 `EXPECTED_ABORT`。

### `latest_if_idle` 的实现原理是什么？

视频采集、处理/编码和 NPU 各在线程中运行。触发帧到来时，只有 NPU `busy=0` 且待处理队列为空才复制当前最新帧给 NPU；否则只记录忙碌跳过。NPU 完成后原子更新最近检测结果，Overlay 对后续视频帧读取并复用它，因此 NPU 不阻塞视频编码。

### 怎么证明没有把预热混进正式数据？

应用和测量边界都使用板端 `CLOCK_MONOTONIC`。CSV 可以包含预热原始行，分析器只选择 metadata 中 `[start,end)` 的正式窗口；这样既可审计预热，又不会污染统计。

### 为什么客户端 FPS 有窗口 FPS 和活动 FPS？

窗口 FPS 用 FFmpeg 总帧数除以固定测试墙钟，包含连接和首帧等待；活动 FPS 用帧数除以解码 PTS 跨度，描述开始收到画面后的节奏。两者都不是客户端显示端到端延迟。

### 如何处理异常和遗留进程？

执行器保存应用、采集器和 FFmpeg 的 PID，所有退出都进入同一个清理路径，先 TERM，超时再 KILL，并回收已有文件。未知的既有 IMX415 进程默认只报告，明确使用 `--stop-existing` 才停止，避免误杀。

### 如何验证统计脚本本身？

用人工构造的 CSV 指定已知帧数、时长、一次视频 drop、一次推理 skip 和确定的阶段时间，断言 FPS、丢帧率、跳过率、P95、NPU均值和完整性结果；再用缺文件样本确认不能误报 Pass。最后用已完成 S0 三轮回放，与旧独立统计交叉核对。

## 排障顺序

1. 先看 `metadata.json` 的测量边界、二进制 MD5、设备模式和失败原因；
2. 看 `stderr.log`、`collector_stderr.log` 和 `ffmpeg_stderr.log` 是否有进程/解码错误；
3. 看 `summary.json.validation` 是时长、样本、时间顺序还是阈值失败；
4. 回到两个原始 CSV，用 Frame ID 和单调时间定位具体行；
5. 确认没有把 AI 忙碌跳过当视频丢帧，没有把旧框复用当新检测；
6. 最后才修改场景阈值或业务代码，并保留修改前备份、差异和新二进制 MD5。
